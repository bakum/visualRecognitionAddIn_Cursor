# VisualRecognition — нативная внешняя компонента 1С

Windows DLL для платформы **1С:Предприятие**: структурирование первичных документов (счёт, счёт-фактура и т.п.) в **JSON (UTF-8)** через **Google Gemini** или **Anthropic Claude** (выбор свойством **`AiProvider` / `ПровайдерИИ`**; для Gemini — ключ **AI Studio**, для Anthropic — ключ **Anthropic API**). Поддерживаются **сканы** и PDF с текстовым слоем. Локального извлечения текста из PDF (MuPDF/PDFium) и методов «только текст» в сборке **нет** — нужен исходящий **HTTPS**.

Лицензия исходного кода компоненты: **GNU AGPL v3** (см. заголовки файлов).

## Возможности

- **Разбор PDF** (`РазобратьПервичныйДокументPdf*`, `РазобратьПервичныйДокументPdfИИ*`): запрос к выбранному провайдеру (**Gemini** или **Anthropic** по `ПровайдерИИ`); BLOB или Base64.
- **Разбор изображения** (`РазобратьПервичныйДокументИзображениеИИ*`): одно растровое изображение (JPEG, PNG, GIF, WebP, BMP, TIFF по сигнатуре). **PDF сюда не передавать** — только методы с `Pdf`. Провайдер — тот же, что для PDF.
- Имя класса: **`VisualRecognition`**.

## Сборка (кратко)

- **CMake** ≥ 3.20, **MSVC**, **C++17**.
- Задайте **`BOOST_ROOT`** на каталог Boost (в `CMakeLists.txt` по умолчанию путь-пример; переопределите `-DBOOST_ROOT=...`).
- Соберите под нужную разрядность (x86 / x64). Имена выходных файлов: `visualAddInWin32.dll`, `visualAddInWin64.dll`.
- Зависимости: **Boost.JSON**, **WinHTTP**; отдельный **pdfium.dll** / **MuPDF** не требуются.

Артефакты для публикации описаны в **`MANIFEST.xml`**.

| `LastGeminiUsageJson` | `ПоследняяСтатистикаТокеновJSON` | Строка | JSON-объект со статистикой токенов последнего вызова Gemini (`promptTokenCount`, `candidatesTokenCount`, `totalTokenCount`) или `{}` если данные не пришли. |

### Коды `ПоследняяОшибка` (`LastErrorCode`)

| Код | Значение |
|-----|----------|
| `0` | Успех. |
| `1` | Пустые данные PDF или изображения. |
| `4` | Некорректная строка Base64. |
| `5` | Неверный тип аргумента: ожидались двоичные данные (BLOB). |
| `6` | Неверный тип аргумента: ожидалась строка UTF-8 (Base64 для PDF/изображения или текст для `GenerateGeminiText`). |
| `7` | Не задан ключ API (`КлючAPIAIStudio` / `AIStudioApiKey`). |
| `8` | Ошибка Gemini (сеть, HTTP, ответ API). |
| `9` | Пустой текст запроса (для `GenerateGeminiText` / `СгенерироватьТекстИИ`). |
| `10` | Текст запроса превышает лимит (~2 МиБ UTF-8). |
| `11` | Неверный тип аргумента: ожидалась строка UTF-8 для простого шифрования/дешифрования текста. |
| `12` | Некорректный формат шифрованного текста (ожидается HEX-строка). |
| `99` | Прочая ошибка; детали — в `ТекстПоследнейОшибки` / поле `error` JSON. |

## Методы

Платформа отдаёт имя в зависимости от языка компоненты: **англ.** (`lang=0`) / **рус.** (`lang=1`).

| Англ. (`lang=0`) | Рус. (`lang=1`) | Описание |
|------------------|-----------------|----------|
| `ParsePrimaryDocumentPdf` | `РазобратьПервичныйДокументPdf` | Gemini; PDF как **BLOB** (совместимость со старым кодом). |
| `ParsePrimaryDocumentPdfBase64` | `РазобратьПервичныйДокументPdfBase64` | То же; PDF в **Base64** (UTF-8). |
| `ParsePrimaryDocumentPdfGemini` | `РазобратьПервичныйДокументPdfИИ` | То же; PDF как BLOB. |
| `ParsePrimaryDocumentPdfGeminiBase64` | `РазобратьПервичныйДокументPdfИИBase64` | То же; Base64. |
| `ParsePrimaryDocumentImageGemini` | `РазобратьПервичныйДокументИзображениеИИ` | Одно изображение как BLOB. |
| `ParsePrimaryDocumentImageGeminiBase64` | `РазобратьПервичныйДокументИзображениеИИBase64` | То же; Base64. |
| `EncodeToBase64` | `КодироватьВBase64` | Кодирование BLOB в строку Base64 (UTF-8). |
| `DecodeFromBase64` | `ДекодироватьИзBase64` | Декодирование строки Base64 (UTF-8) в BLOB. |
| `EncryptTextSimple` | `ШифроватьТекстПросто` | Простое симметричное шифрование UTF-8 строки; результат — HEX-строка. |
| `DecryptTextSimple` | `РасшифроватьТекстПросто` | Обратное преобразование HEX-строки, полученной `EncryptTextSimple`, в исходный UTF-8 текст. |
| `EncodeKey` | `КодироватьКлюч` | Упаковка ключа: UTF-8 строка → `EncodeToBase64` → `EncryptTextSimple`; возвращает HEX-строку. |
| `DecodeKey` | `ДекодироватьКлюч` | Обратная распаковка ключа: `DecryptTextSimple` → `DecodeFromBase64` → исходная UTF-8 строка. |
| `GenerateGeminiText` | `СгенерироватьТекстИИ` | Один аргумент — **строка UTF-8** (промпт); ответ без схемы первички — **обычный текст** модели. |
| `GetSupportedGeminiModels` | `ПолучитьПоддерживаемыеМоделиGemini` | Без параметров: JSON каталога моделей (**2.5+**, превью **3.x**). |
| `UseFastGeminiTimeoutsProfile` | `ИспользоватьБыстрыйПрофильТаймаутовGemini` | Без параметров: установить быстрый профиль таймаутов (`GeminiReceiveTimeoutMs=20000`, `GeminiTotalDeadlineMs=30000`). Возвращает JSON с применёнными значениями. |

**Возвращаемое значение:** для методов разбора первички — строка UTF-8: JSON от модели или `{"error":"..."}`; для `GenerateGeminiText` / `СгенерироватьТекстИИ` при успехе — **произвольный текст** ответа, при ошибке — `{"error":"..."}`.

При кодах **`5`** и **`6`** дополнительно выбрасывается **исключение** (неверный тип параметра).

После вызовов Gemini (`РазобратьПервичныйДокумент*`, `РазобратьПервичныйДокументИзображениеИИ*`, `GenerateGeminiText`) статистика токенов доступна в свойствах `LastPromptTokens` / `LastOutputTokens` / `LastTotalTokens` и в `LastGeminiUsageJson`. Для отладки сырого ответа используйте `LastGeminiRawResponsePreview` / `ПоследнийСыройОтветGeminiПревью`.

## Формат успешного JSON

Задаётся промптом к Gemini; ориентировочно: `counterparty`, `contract`, `invoiceNumber`, `documentDate`, `bankDetails`, `lineItems`, `rawText` и др. Точный набор полей зависит от ответа модели.


## Обработка ошибок

1. Проверяйте **`ПоследняяОшибка` = 0**.
2. При ненулевом коде показывайте **`ТекстПоследнейОшибки`**.
3. В строке результата может быть JSON с полем `error` — согласуйте с кодом в свойствах.

## История версий

Номер версии задаётся в **`CMakeLists.txt`** (`project(... VERSION ...)`) и доступен в 1С в свойстве **`Версия`** / **`Version`**. В репозитории для релизов используются теги вида **`v2.0.12`**.

### 2.0.12

- **Провайдер ИИ:** свойство **`AiProvider` / `ПровайдерИИ`** — `gemini` (по умолчанию), `anthropic`, `claude`, а также кириллица **`антропик`**, **`клод`** (варианты регистра). Один и тот же выбор применяется к разбору первички (PDF/изображение) и к **`GenerateGeminiText` / `СгенерироватьТекстИИ`**.
- **Anthropic:** свойства **`AnthropicApiKey` / `КлючAPIAnthropic`**, **`AnthropicModel` / `МодельAnthropic`**, **`AnthropicMaxOutputTokens` / `МаксТокеновВыводаAnthropic`**, превью ответа **`LastAnthropicRawResponsePreview` / `ПоследнийСыройОтветAnthropicПревью`**; метод каталога **`GetSupportedAnthropicModels` / `ПолучитьПоддерживаемыеМоделиAnthropic`**; PDF через Messages API с заголовком **`anthropic-beta: pdfs-2024-09-25`**.
- **Защита от путаницы моделей:** если в **`МодельGemini`** указан идентификатор Claude (или наоборот), возвращается явная ошибка с подсказкой переключить провайдер/свойство модели.
- **Сообщения о пустом ключе:** для Gemini и Anthropic используются **разные** внутренние коды ошибки (`Gemini API key is empty` / `Anthropic API key is empty`), чтобы в **`ТекстПоследнейОшибки`** не подставлялся текст про AI Studio при выбранном Anthropic и пустом ключе Anthropic.
- **Распознавание значения провайдера:** обрезка UTF-8 BOM и неразрывного пробела по краям; для латиницы допускаются значения с префиксом **`anthropic…`** / **`claude…`** после нормализации буквенно-цифрового ключа; если в варианте свойства **нет строки**, используется ветка Gemini (как при значении по умолчанию).
- **Исправление (Anthropic + изображения):** MIME типа изображения передавался в API через **`string_view`**, указывающий на уже уничтоженную строку из **`optional`** — для растровых форматов возникала ложная ошибка **`Invalid inline MIME type`**; теперь MIME хранится в **`std::string`** до вызова Anthropic API.
